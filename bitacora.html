Me estoy construyendo

Esto no va solo sobre una web. Va sobre lo que pasa cuando alguien no técnico empieza a usar Codex, Git y GitHub para construir algo propio y, en el camino, descubre que muchas herramientas dejan de intimidar cuando por fin tocan un problema real.

Publicación

Lo que estoy aprendiendo mientras hago este sitio

Esta página me interesa más como registro de aprendizaje que como una lista de cambios del proyecto. Cuando empecé, muchas de estas herramientas se sentían lejanas o demasiado técnicas. Codex, Git, GitHub, los workflows, el deploy, incluso la idea de tocar archivos con intención. Todo parecía pertenecer a otro mundo.

Lo que cambió no fue solo el sitio. Cambió sobre todo la forma en la que me relaciono con ese mundo. En vez de mirar conceptos sueltos desde afuera, empecé a entenderlos mientras hacía algo propio. Esa diferencia fue clave. Aprender dejó de ser una tarea abstracta y empezó a parecerse más a resolver problemas concretos con contexto real.

Codex apareció primero como una herramienta para avanzar más rápido, pero en la práctica me resultó más valioso como compañero de iteración. Sirve mucho cuando puede leer lo que ya existe, detectar fricciones, proponer cambios concretos y explicar qué está haciendo. Ahí deja de sentirse como una cosa mágica y pasa a sentirse como una forma distinta de pensar en voz alta con asistencia.

También empecé a entender mejor Git. Antes podía verse como una capa innecesaria de complejidad. Ahora se parece más a una manera de ordenar el trabajo. Guardar cambios, leer diferencias, pensar commits con intención, probar sin romper todo. No hace falta dominarlo entero para que empiece a ser útil. Alcanza con ver que tiene consecuencias concretas sobre cómo uno trabaja y sobre la tranquilidad con la que puede iterar.

Con GitHub pasó algo parecido. Al principio era fácil pensarlo solo como el lugar donde vive el repositorio. Después empezó a volverse más claro que también es publicación, historial, validación y contexto operativo. Ver el sitio publicado en Pages, entender que hay checks, workflows y despliegues, hace que todo eso deje de sonar lejano. El proyecto ya no es solo una carpeta: es un sistema pequeño que vive y se actualiza.

CI/CD fue quizás una de las cosas que más cambió de significado. Desde afuera suena técnico y abstracto. Desde adentro puede traducirse de forma bastante simple: antes de publicar, conviene que algo revise si lo básico sigue funcionando; después, si está bien, se publica. Entenderlo así le saca bastante dramatismo y ayuda a ver que no se trata de infraestructura por la infraestructura misma, sino de reducir errores y dar más confianza.

Lo más interesante de todo esto es que el aprendizaje no vino de estudiar cada herramienta por separado. Vino de usarlas sobre algo propio. Esa mezcla entre proyecto personal, práctica concreta y acompañamiento de Codex fue lo que más ayudó a bajar barreras. Muchas cosas que parecían reservadas para gente técnica se vuelven mucho más accesibles cuando están ligadas a una necesidad real.

Me gustaría que esta página siga creciendo así: como una publicación viva, más cercana a una nota de trabajo que a una vitrina. Un lugar para registrar qué entendí, qué me costó, qué preguntas aparecieron y qué partes del proceso empezaron a hacer sentido recién después de tocar el proyecto varias veces.

  • Codex ayuda más cuando trabaja sobre contexto real y explica lo que hace.
  • Git empezó a tener sentido cuando dejó de ser teoría y pasó a ordenar decisiones concretas.
  • GitHub se volvió más claro cuando apareció como parte del producto y no solo como repositorio.
  • CI/CD dejó de sonar abstracto cuando pude traducirlo a validación antes de publicar.
  • Aprender desde un proyecto propio baja mucho la barrera para alguien no técnico.
